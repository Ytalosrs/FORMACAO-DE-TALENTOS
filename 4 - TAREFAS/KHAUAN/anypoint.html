Definição

O que a plataforma entrega

A Anypoint Platform funciona como o hub central do ecossistema MuleSoft e como a infraestrutura da Application Network.

Ela conecta sistemas, integrações e automações, permitindo que Flow, Composer, RPA e Mule apps compartilhem APIs e assets governados.

🧠

Backbone técnico

  • Integração centralizada.
  • Governança e segurança aplicadas de forma consistente.
  • Reuso como princípio arquitetural, não como exceção.

Arquitetura

API-led connectivity

Camada 1

System APIs

Conectam aos sistemas de record e isolam consumidores das mudanças do backend.

Camada 2

Process APIs

Orquestram múltiplas fontes e encapsulam lógica de negócio reutilizável.

Camada 3

Experience APIs

Adaptam a entrega para cada canal, consumidor ou jornada.

Ponto-chave: quando vários projetos usam o mesmo sistema, a System API é reutilizada em vez de recriada, aumentando padronização e velocity.

Reuso

Composable building blocks

O que são

  • APIs, specs e templates pensados para reaproveitamento.
  • Blocos intercambiáveis que reduzem integrações ponto a ponto.
  • Base para soluções montadas por composição.

Como aparecem na prova

  • Assets publicados no Exchange viram insumos para novos projetos.
  • Flow, Composer, RPA e Mule apps podem consumir a mesma base de APIs.
  • Reuso é tratado como ganho de arquitetura e entrega.

API Manager

Basic endpoint vs proxy endpoint

Basic endpoint

  • Gerencia a API diretamente.
  • Define a inbound URL monitorada no API Manager.
  • Útil quando a integração já está preparada para gestão direta.

Proxy endpoint

  • Insere um intermediário entre cliente e backend.
  • Aplica policies antes de encaminhar a chamada.
  • Pode proteger backends Mule ou não Mule.

Na prova, a distinção mais importante é reconhecer quando o proxy é necessário para centralizar políticas e governança sem alterar o backend.

Policies

O que as políticas resolvem

🔐

Segurança

OAuth 2.0, JWT, Basic Auth, Client ID Enforcement.

🚦

Tráfego

Rate limiting, throttling e SLA-based.

🌐

Acesso

IP allowlist e blocklist por origem.

⚙️

Qualidade

Caching, timeout e custom policies.

Execução

As policies são configuradas no API Manager, mas executadas internamente no proxy ou app.

Governança

Mudanças podem ser distribuídas sem redeploy; automated policies reforçam padrão por ambiente.

Visibilidade

Private vs Public Portal

Private Portal

  • Visível apenas dentro da organização.
  • Mais alinhado a APIs internas e dados sensíveis.
  • Base da Application Network interna.

Public Portal

  • Exposição controlada para parceiros e terceiros.
  • Amplia a rede de integração para fora da empresa.
  • Exige governança forte e uso correto de policies.

Regra mental útil: System APIs tendem a ser private; Process APIs geralmente também; Experience APIs podem ser public quando há integração externa.

Memorização

O que vale lembrar para a certificação

Perguntas recorrentes

  • Quando usar CloudHub, RTF ou hybrid.
  • Diferença entre basic endpoint e proxy endpoint.
  • Como o Exchange sustenta reuso e governança.

Armadilhas comuns

  • Confundir visibilidade pública com boa prática universal.
  • Esquecer que policies não exigem mudança de código.
  • Ignorar que API Monitoring depende de Autodiscovery.
